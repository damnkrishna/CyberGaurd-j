## What I Now Know End-to-End

I understand, inside and out, how a SIEM actually works:

- **Data Ingestion** — how raw logs flow in from different sources
- **Data Normalization** — how those logs get standardized into a consistent schema (OCSF in our case)
- **Detection Logic** — how correlation rules sit on top of that normalized data and fire on real threats
- **SOC Analyst Workflow** — how an analyst actually triages and resolves alerts on the other end

---

## The False Positive Problem — And How I Fixed It

A colleague tried writing custom correlation rules but kept the scope too broad, which resulted in too many false positives. That's not right — a noisy SIEM is almost as bad as no SIEM.

So I stepped in and rewrote the rules to be **precise, not broad** — rules that actually target the real problem and generate fewer false positives.

For every rule I've written, I've documented:

- The detection logic (Sigma format)
- OCSF-normalized Sigma code
- What false positives it can potentially generate
- How to handle or suppress those false positives

All of it is on my GitHub, documented properly.
