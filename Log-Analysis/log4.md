# Mastering Linux Logging & Log Analysis

A critical skill for any Security Operations Center (SOC) analyst is the ability to navigate and interpret system logs. Today, I explored the transition from traditional plaintext logs to the powerful, centralized **systemd** logging ecosystem used in modern Linux distributions like **Kali Linux**.

Understanding how a system records its history is the first step toward detecting an intruder, investigating a crash, or validating an access attempt.

---

##  The Modern Logging Architecture

In modern Linux distributions, logging has evolved from simple text files managed by `syslog` into a robust, structured database managed by **systemd**. This system offers better indexing, faster searching, and protection against casual log tampering.

### Key Components

* **systemd**: The overall system and service manager that handles the entire OS lifecycle, including the initialization of logging services. It does not care where a log originates; it ensures the infrastructure is there to catch it.
* **journald**: The specific **daemon** (background service) that collects data from the kernel, system services, and applications. It stores this data in a secure **binary format**.
* **journalctl**: The powerful command-line utility used to query, display, and analyze the logs collected by `journald`.

> **SOC Analyst Thought:** The binary nature of `journald` logs is a significant security feature. While an attacker can easily use an editor like `nano` to delete their tracks from a traditional plaintext file like `/var/log/auth.log`, altering the indexed `journald` binary database requires much higher-level access and specialized tools, leaving a "trace" of its own.

---

## 🛠️ The `journalctl` Toolbox

Since `journalctl` interacts with a structured database, we use specific flags to filter out the "noise" and find the security events we care about.

### Filtering by Time & Boot

* `journalctl -b`: View logs from the **current boot** only.
* `journalctl -b -1`: View logs from the **previous boot**. (Essential for forensic analysis after a system crash).
* `journalctl --since "1 hour ago"`: Limit results to a specific timeframe.
* `journalctl --until "2026-04-03 20:00:00"`: View logs up to a specific moment.

> **TIP:** For quick time-based searches, you can combine these flags. A SOC analyst might use `journalctl --since "12:00" --until "13:00"` to pinpoint exactly what happened during a reported security incident.

### Filtering by Source

* `-u <service>`: Filter by a specific systemd unit (e.g., `ssh` or `apache2`).
* `-k`: View **Kernel** logs only. (Crucial for identifying hardware failures or kernel-level rootkits).
* `_PID=<ID>`: Filter logs created by a specific Process ID.
* `_UID=<ID>` / `_GID=<ID>`: Filter logs created by a specific User or Group.

> **SOC Analyst Thought:** The `-u <service>` flag is the most important for application monitoring. If you suspect an attacker is attempting a Brute Force attack, your first command should be `sudo journalctl -u sshd -f`.

### Real-time & Priority

* `-f`: **Follow** the logs in real-time. (Standard practice for monitoring an attack or a service as you interact with it).
* `-p <level>`: Filter by **Priority**. The levels range from `0` (emerg) to `7` (debug). Filtering by `err` (3), `warning` (4), or `crit` (2) allows you to bypass normal operational logs and focus on system failures.

##  Practical Lab: SSH Brute-Force Analysis

To put theory into practice, I simulated a common attack scenario on my Kali machine.

### Mission
Identify the logs generated by "Failed Password" attempts for the SSH service.

### Steps Taken
1.  **Service Activation**: Started the SSH service using `sudo systemctl start ssh`.
2.  **Verify Status**: Validated the service was running and listening on port 22 using `sudo systemctl status ssh`.
3.  **Attack Simulation**: Attempted to SSH into `localhost` using invalid credentials multiple times from a second terminal window.
4.  **Log Retrieval**: Used `sudo journalctl -u ssh -f` to watch the authentication failures in real-time.

### Findings
I successfully identified the "Failed password" entries. This single log line contains all the critical data points required by a professional SIEM (Security Information and Event Management) system:
* **Timestamp**: When the attempt happened (`Apr 03 19:58:35`).
* **Source IP**: Where the attack came from (`127.0.0.1`).
* **Username**: The account being targeted (`invalid user jess`).

---

> **Pro-Tip:** Remember that on "volatile" distros like Kali Linux, logs may be kept only in RAM (`/run/log/journal`). If you need logs to persist after a reboot, you must explicitly configure `journald` to store them on the disk (`/var/log/journal`) by editing the `/etc/systemd/journald.conf` file.
